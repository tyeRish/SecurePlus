CRYPTOGRAPHIC HASHING — LIVE DEMO
A hash function takes any input and produces a fixed-length output. The same input always produces the same output. It's a one-way operation — you cannot reverse a hash to get the original data.

PASSWORD SALTING DEMO
If two users have the same password, their hashes are identical — making rainbow table attacks easy. Salting adds a unique random value to each password before hashing, making all hashes unique.
Why this matters: If an attacker gets your hash database, identical hashes immediately reveal who shares passwords. Unique salts prevent this and defeat precomputed rainbow tables.
HASH ALGORITHM COMPARISON
  • ✓
    SHA-256 — 256-bit output. Current standard. Used in TLS, certificates, code signing. Computationally fast — use bcrypt/Argon2 for passwords.
  • ✓
    SHA-3 — 256/384/512-bit. Different internal design (Keccak sponge) from SHA-2. Backup standard in case SHA-2 weaknesses found.
  • ✗
    SHA-1 — 160-bit. Cryptographically broken (collision attacks demonstrated). Deprecated in certificates since 2017.
  • ✗
    MD5 — 128-bit. Completely broken. Collision attacks trivial (find two inputs with same hash). Never use for security. Still used for non-security checksums.
  • ⊕
    bcrypt / Argon2 — Intentionally SLOW password hashing algorithms. Add salt automatically. Configurable work factor — tune to stay expensive as hardware improves.
AES ENCRYPTION — LIVE DEMO
AES (Advanced Encryption Standard) uses the same key to encrypt and decrypt. It operates on 128-bit blocks. AES-256 uses a 256-bit key — the current gold standard for symmetric encryption.

How AES-GCM works: The key is derived using PBKDF2. A random 96-bit IV (Initialization Vector) is generated for each encryption — the same key + different IV = completely different ciphertext. The IV is transmitted alongside the ciphertext (it's not secret, just unique). GCM mode also provides an authentication tag — any tampering with the ciphertext is detected.
AES MODES OF OPERATION
  • ✗
    ECB (Electronic Codebook) — Broken. Each block encrypted independently. Same plaintext = same ciphertext. Patterns visible in encrypted data. Never use.
  • △
    CBC (Cipher Block Chaining) — Each block XORed with previous ciphertext before encryption. Better than ECB, but requires padding, vulnerable to padding oracle attacks. Removed from TLS 1.3.
  • ✓
    GCM (Galois/Counter Mode) — Stream mode (no padding needed) + authentication tag. Provides confidentiality AND integrity (AEAD). Mandatory in TLS 1.3. Recommended.
  • ✓
    CTR (Counter Mode) — Stream mode. Converts AES block cipher to keystream generator. No padding. Parallelizable. No authentication (add HMAC separately).
ECB WEAKNESS VISUALIZATION
ECB mode encrypts each 16-byte block independently. This makes patterns visible. A famous example is an encrypted Linux Tux penguin image — the shape remains visible even after ECB encryption.
RSA KEY GENERATION — LIVE DEMO
RSA uses a mathematically linked key pair: the public key encrypts (or verifies), the private key decrypts (or signs). Generate a real RSA-2048 key pair right here in your browser.
ENCRYPT WITH PUBLIC KEY
Anyone with the public key can encrypt data. Only the holder of the private key can decrypt it. This is how asymmetric encryption works for confidentiality.
DECRYPT WITH PRIVATE KEY
The encrypted message can only be decrypted with the corresponding private key. The private key never leaves its owner.
RSA vs AES — WHEN TO USE WHICH
  • ✓
    RSA for key exchange — Asymmetric is ~1000× slower than AES. Used only to securely exchange a symmetric key, not to encrypt data directly.
  • ✓
    AES for data — Once both parties have the shared symmetric key (exchanged via RSA or ECDH), all data is encrypted with fast AES.
  • →
    TLS combines both — RSA/ECDH for handshake, AES-GCM for all subsequent data. Best of both worlds.
  • ✍
    RSA for digital signatures — Sign with private key (only you can sign). Verify with public key (anyone can verify). Provides authentication and non-repudiation.
KEY SIZE AND SECURITY
  • ✗
    RSA-512 / RSA-1024 — Broken / deprecated. Factored by researchers. Not acceptable for any modern use.
  • △
    RSA-2048 — Current minimum acceptable standard. Estimated secure until ~2030. Still widely used.
  • ✓
    RSA-3072 / RSA-4096 — Recommended for long-lived keys and sensitive applications. Significantly slower.
  • ✓
    ECDSA P-256 / P-384 — Elliptic curve equivalent to RSA-3072/RSA-7680. Much smaller keys, faster operations. Preferred in modern systems.

ECDH KEY EXCHANGE — LIVE DEMO
ECDH (Elliptic Curve Diffie-Hellman) is the modern version used in TLS 1.3. Alice and Bob each generate a key pair. They exchange public keys and both independently compute the same shared secret. The secret never crosses the network.
PERFECT FORWARD SECRECY (PFS)
With static RSA key exchange: an attacker who records all encrypted traffic now AND later steals the private key can decrypt everything retroactively. ECDHE (Ephemeral) generates a fresh key pair for every session.
  • ✗
    Static RSA (no PFS) — One private key encrypts all sessions. Compromise the private key = decrypt all past and future traffic. Removed in TLS 1.3.
  • ✓
    ECDHE (Ephemeral) — Fresh keypair per session. Compromise one session key = one session's data. Past sessions remain secure. Provides PFS.
  • ✓
    TLS 1.3 mandates ECDHE — All TLS 1.3 handshakes use ephemeral key exchange. PFS is not optional — it's built in.

TLS 1.3 vs TLS 1.2 — KEY DIFFERENCES
  • ✓
    1-RTT handshake (TLS 1.3) vs 2-RTT (TLS 1.2). 0-RTT available for session resumption (with replay attack trade-off).
  • ✓
    Mandatory ECDHE — PFS built in. TLS 1.2 allowed static RSA (no PFS).
  • ✓
    Encrypted extensions — In TLS 1.3, most of the handshake metadata is encrypted. TLS 1.2 sent extensions in plaintext.
  • ✓
    Removed weak ciphers — No RC4, DES, 3DES, MD5, SHA-1, CBC mode, RSA key exchange, or compression.
  • ✓
    Only 5 cipher suites — vs 37+ in TLS 1.2. All use AEAD (AES-GCM or ChaCha20-Poly1305).
CERTIFICATE CHAIN — HOW TRUST WORKS
  • 1
    Root CA — Self-signed certificate from a trusted root authority (DigiCert, Let's Encrypt). Stored in OS/browser trust store. Never used directly to sign leaf certs.
  • 2
    Intermediate CA — Signed by Root CA. Issues end-entity (leaf) certificates. If compromised, only revoke the intermediate — not the root.
  • 3
    Leaf Certificate — Your website's certificate. Contains: domain name, public key, validity period, issuer (Intermediate CA), digital signature.
  • ✓
    Validation — Browser verifies each signature up the chain. All certs must be valid (not expired, not revoked, correct domain). Chain must terminate at a trusted Root CA.
